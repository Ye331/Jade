# Jade 项目协作宪章

## 项目定位

`Jade` 是一个 Unity 2D 类银河恶魔城平台跳跃游戏。核心体验参考《奥日与黑暗森林》：流畅的移动手感、富有弹性的空中控制、能力解锁驱动的地图回访、环境叙事，以及由音乐、光影和关卡节奏共同塑造的情绪推进。

项目的主题方向是中式神话与水墨剪影。我们不直接复刻某一段传统神话，而是提炼“山海、灵玉、残庙、幽林、天河、冥府、妖灵、神迹崩塌”等意象，构建一个失落神域逐渐苏醒的世界。核心意象围绕“玉、灵、破碎秩序、重铸与归还”展开。

首个可交付目标不是完整剧情切片，而是移动手感原型。平台跳跃游戏的地基是手感，后续的关卡、美术、战斗和叙事都要服务于“玩家愿意只是移动也觉得舒服”这一点。

这个项目同时是课程作业，开发者也是初学者。因此所有计划都要控制复杂度，优先做能按时展示、能讲清楚实现思路、能体现学习成果的版本。功能宁可少而完整，也不要铺太大导致无法收尾。

## 长期协作原则

- 每一轮工作都优先交付可验证结果：能跑、能跳、能看、能测、能调整。
- 面向初学者协作：代码要直观、命名要清楚、拆分要适度，避免为了“高级”而引入难懂架构。
- 程序实现采用 Unity C# 组件化思路，避免过早做庞大框架；当重复和复杂度真实出现时再抽象。
- 玩家手感、相机、碰撞、状态机、能力系统分层实现，便于调参和替换。
- 美术先走水墨剪影原型路线，用纯色块、SpriteShape、Tilemap、粒子、光源、后处理和简单贴图快速验证气质，再逐步替换为精制资产。
- 每个阶段都要有可玩验收标准，不只交付代码、素材或文字设定。
- 我会持续参与程序代码、关卡灰盒、美术提示词、资产规格、测试用例、重构建议、Bug 定位和文档维护。
- 任何新增系统都应尽量同步提供最小验证场景、测试步骤或手动验收说明。
- 课程作业优先级高于商业化完整度：遇到取舍时，优先保证可演示、可解释、可提交。
- 原型阶段允许丑，但不能含糊；正式化阶段允许慢，但要可维护。

## 学习与作业约束

- 每个阶段都要留下能用于汇报的材料：目标、实现方式、截图或录屏点、遇到的问题和解决方式。
- 新增核心脚本时，优先写清楚职责边界；必要时加少量注释帮助复习，但不要把注释写成噪音。
- 功能实现采用“最小可玩版本 -> 调参 -> 再扩展”的顺序，避免一开始同时做移动、战斗、存档、地图和美术全套。
- 对初学者不友好的方案需要主动降级，例如复杂状态机框架、第三方大型插件、过度泛化的数据驱动系统。
- 每次阶段结束都要能回答三个问题：做了什么、为什么这样做、下一步还差什么。

## 工程约定

当前工程是 Unity/团结 2D 项目，编辑器版本见 `ProjectSettings/ProjectVersion.txt`：Unity 2022.3.61t11 / Tuanjie 1.6.10。项目已经从空工程推进到可运行的 2D 移动与灰盒关卡原型。

当前 Build Settings 中包含：

- `Assets/Scenes/SampleScene.scene`
- `Assets/Scenes/Prototype/Phase01_BasicMovement.scene`
- `Assets/Scenes/Prototype/Phase02_GrayboxLevel.scene`

推荐的目录方向：

- `Assets/Scripts/Player`：玩家输入、移动、能力、生命、动画状态。
- `Assets/Scripts/World`：关卡机关、能力门、检查点、区域触发器。
- `Assets/Scripts/Combat`：伤害、受击、敌人 AI、命中盒。
- `Assets/Scripts/Camera`：跟随、边界、镜头区域、震屏。
- `Assets/Scenes/Prototype`：手感、战斗、能力门控等验证场景。
- `Assets/Art/PrototypeArt`：临时图形、测试贴图、灰盒素材。
- `Assets/Art/Characters`：角色正式素材。
- `Assets/Art/Environments`：场景正式素材。
- `Assets/Art/VFX`：粒子、特效、后处理资源。
- `Assets/Design`：策划表、参数记录、关卡草图、验收记录。

输入方案先使用现有 Unity 旧 InputManager 启动原型。等移动、战斗和 UI 需求稳定后，如果需要手柄重绑定、多设备输入或更完整的输入配置，再迁移到 Unity Input System。

原型资产允许使用纯色块、SpriteShape、Tilemap、粒子和简单水墨贴图。正式资产需要记录尺寸、pivot、动画帧率、命名规则和用途，避免替换素材时破坏碰撞、动画或镜头构图。

- 视觉方向已从纯方块代理推进到玉麒麟临时帧与 Animator 管线：Idle、Run、Turn、JumpStart、JumpRise、JumpApex、Fall、Land。
- 背景已接入 `JadeMythForest_Background`，作为相机子物体的远景氛围层，用于验证中式神话水墨剪影气质。

### 已实现脚本职责

- `PlayerInputReader`：读取旧 InputManager 的 Horizontal 与 Jump 输入，并提供按下/松开消费接口。
- `PlayerMotor2D`：负责 Rigidbody2D 移动、跳跃、重力、落地检测、朝向、出生点和复活点。
- `PlayerMovementSettings`：集中保存移动手感参数，当前主配置为 `Assets/Scripts/Player/LightweightMovement.asset`。
- `PlayerVisualFeedback2D`：负责视觉根节点翻转、前倾、拉伸压缩、落地尘土和跑动尘土，不改变物理碰撞。
- `PlayerAnimationDriver2D`：把移动状态写入 Animator 参数，并根据跑速轻微调整 Run 播放速度。
- `SimpleCameraFollow2D`：负责相机平滑跟随、前视偏移和简单震屏接口。
- `PrototypeBasicMovementSceneBuilder`：运行时生成 Phase01 手感测试场景。
- `PrototypeGrayboxLevelSceneBuilder`：运行时生成 Phase02 灰盒关卡、检查点、收集物、终点和掉落复位区。
- `Checkpoint2D`：触发后更新玩家复活点，并改变检查点显示颜色。
- `Collectible2D`：提供最小收集反馈，当前只记录静态计数并关闭对象。
- `LevelGoal2D`：提供关卡结束触发和颜色反馈。
- `RespawnZone2D`：玩家掉落到触发区后回到当前出生点。

### 美术与动画管线

- 临时低清帧位于 `Assets/Resources/Characters/JadeQilinFrames/`，用于当前原型快速验证。
- 正式高分辨率帧预留在 `Assets/Resources/Characters/JadeQilinHighResFrames/`，规格见 `Assets/Design/JadeQilin_AnimationImportSpec.md`。
- 编辑器菜单 `Jade > Build Jade Qilin Player Assets` 会重建动画片段、Animator Controller 和玩家 prefab。
- `JadeQilinSpriteImportPostprocessor` 会自动为玉麒麟帧图应用 2D Sprite、底部中心 pivot、无压缩、无 mipmap、512 PPU 等导入设置。

### 当前范围说明

- Phase01 当前实际交付的是“基础移动手感原型”，暂不包含墙滑、墙跳、冲刺、二段跳、攀爬、战斗、生命系统和正式 UI。
- Phase02 当前实际交付的是“灰盒路线验证”，暂不包含能力门、持久化存档、敌人、正式地图 UI 和正式关卡美术。
- 当前检查点只更新本次 Play Mode 内的复活位置，不写入磁盘存档。
- 当前收集物只做最小反馈，不保存收集状态，也不驱动奖励系统。

## 分阶段路线

### Phase 0: 项目骨架

目标：建立可长期扩展的基础工程形态。

主要任务：

- 建立目录结构、命名规范和原型资源存放规则。
- 创建基础测试场景和玩家起点。
- 明确输入方案、相机方案、碰撞层和 Sorting Layer。
- 准备最小可用的原型素材规范。
- 建立简单的测试和验收记录方式。
- 准备课程汇报用的项目说明、阶段截图点和开发日志模板。

验收标准：

- 新增脚本、场景、素材有明确归属目录。
- 能打开一个原型场景并运行基础对象。
- 后续 agent 或开发者能从文档理解项目方向和下一步优先级。
- 初学者能根据目录和文档找到主要脚本、测试场景和下一步任务。

### Phase 1: 移动手感原型

目标：完成第一版“愿意反复跑跳”的玩家控制器。

主要任务：

- 横向移动、加速、减速、空中控制。
- 跳跃、可变跳高、土狼时间、跳跃缓冲。
- 墙滑、墙跳、冲刺。
- 受击回退、短暂无敌、落地恢复。
- 基础相机跟随、前视偏移和简易震屏。
- 建立手感参数表，便于后续持续调参。

验收标准：

- 玩家能在测试场景完成跑、跳、墙滑或墙跳、冲刺、落地恢复。
- 平台边缘跳跃容错明显，跳跃缓冲有效。
- 镜头跟随稳定，不抖动，不遮挡主要前进方向。
- 所有核心移动参数能在 Inspector 或配置资源中调整。

### Phase 2: 探索与能力门控

目标：让移动能力开始服务类银河恶魔城结构。

主要任务：

- 能力解锁与持久化状态。
- 存档点和复活点。
- 区域门、能力门、一次性捷径。
- 房间式地图结构和基础回溯路径。
- 简易 UI 提示，展示获得能力或无法通过的原因。

验收标准：

- 至少一个能力解锁能打开新路径。
- 至少一个存档点能记录玩家位置和已获得能力。
- 灰盒地图存在回环、能力门和回访收益。

### Phase 3: 战斗与敌人

目标：建立轻量但可扩展的战斗闭环。

主要任务：

- 玩家基础攻击、命中检测、伤害数值。
- 玩家生命、受击、无敌帧、死亡与复活。
- 敌人巡逻、追击、攻击、受击、死亡。
- 简单掉落物或奖励反馈。
- 首个小 Boss 原型，用于验证战斗节奏和场地设计。

验收标准：

- 玩家和敌人能互相造成伤害。
- 敌人行为可调参，死亡和受击反馈清楚。
- 小 Boss 原型具备至少两个攻击节奏，并能被玩家击败。

### Phase 4: 中式神话美术垂直切片

目标：完成一个能代表项目气质的区域样片。

主要任务：

- 选择首个主题区域，例如幽林残庙、天河断桥或玉矿神龛。
- 制作前景剪影、中景平台、远景水墨层和视差背景。
- 加入粒子、光源、雾效、环境音和音乐占位。
- 制作主题 UI 的基础风格，如生命、灵力、存档点提示。
- 让美术替换不破坏现有碰撞、路径和镜头。

验收标准：

- 单一区域从截图上能明确读出“中式神话水墨剪影”气质。
- 玩家可以在该区域完成一段完整移动和探索路线。
- 美术层级、碰撞层级和交互对象清晰分离。

### Phase 5: 内容扩展

目标：从垂直切片扩展到可持续制作的内容管线。

主要任务：

- 扩展多个区域主题和区域间连接。
- 设计能力链路，让新能力改变旧区域的探索方式。
- 增加敌人生态、收藏物、剧情碎片和环境叙事。
- 建立关卡灰盒到正式美术替换的流程。
- 定期维护参数表和资产规格。

验收标准：

- 至少两个区域有清晰的视觉和玩法差异。
- 能力回访带来实际路线变化或奖励。
- 内容制作流程可以复用，不依赖临时手工补丁。

### Phase 6: 打磨与发布准备

目标：准备可公开试玩的 Demo。

主要任务：

- 性能检查、资源压缩、加载体验和内存控制。
- 手感参数统一整理，避免散落在场景对象里。
- 存档稳定性、死亡复活、场景切换和异常状态修复。
- 音频、特效、UI、震屏、暂停菜单等体验打磨。
- 建立打包流程、发布检查表和已知问题列表。

验收标准：

- Demo 可以从头到尾稳定游玩。
- 没有阻断流程的严重 Bug。
- 打包流程可重复执行，发布说明清楚。

## 首批优先级

1. 完成 Phase 0 的目录结构和原型场景。
2. 实现 Phase 1 玩家移动控制器。
3. 建立一条专门测试跑跳、墙跳、冲刺和镜头的灰盒路线。

5. 用最少美术资产建立水墨剪影氛围测试，而不是过早制作完整角色和背景。
6. 同步整理课程作业汇报材料：项目简介、目标拆分、关键脚本说明、阶段成果截图和后续计划。

## 测试习惯

- 每次实现玩法功能后，优先确认 Unity 编译无错误。
- 涉及玩家移动、战斗、镜头或关卡时，补充手动 Play Mode 验收步骤。
- 涉及数据持久化时，测试新存档、旧存档、死亡复活和重新进入场景。
- 涉及美术替换时，测试碰撞、排序层级、镜头遮挡和可读性。
- 任何阶段如果出现“感觉不对”，优先回到测试场景调参，而不是继续堆内容。

## 默认假设

- 文档使用中文维护，作为长期项目协作说明。
- 项目是课程作业，最终需要能展示、能讲解、能按时提交。
- 开发者是初学者，所有代码和流程都要尽量清楚、可学习、可调试。
- 项目继续使用当前 Unity/团结 2D 工程，不切换引擎。
- 首个可玩目标是移动手感原型。
- 美术默认走水墨剪影路线，先低成本验证氛围，再逐步精制。
- 暂不引入复杂第三方角色控制器，先做可控、可调、适合本项目的轻量实现。
